# 类游戏角色编辑器路线

目标是把角色生产拆成可调参数：身份、体型、服装、发型、妆容、鞋袜、姿势、场景和画风都能分层控制。

## ComfyUI 参数化路线

ComfyUI 路线使用：

- 角色身份 LoRA：控制“是谁”。
- 服装/鞋袜/发型/妆容参考：控制外观模块。
- IPAdapter：输入图片参考。
- ControlNet：控制姿势、构图和身体比例。
- inpaint：修复脸、眼镜、手、腿、鞋、服装边缘。
- prompt 参数：描述镜头、服饰、风格、场景和情绪。

这种路线适合先做 2D 漫画和角色设定图，因为成本低、迭代快、能直接复用现有 ComfyUI 工作流。

## 3D 角色基底 + ComfyUI 风格化路线

后续可以用 3D 角色作为统一体型和姿势底座，再用 ComfyUI 做风格化。

适用场景：

- 长篇漫画需要稳定身高、腿长、体型和镜头角度。
- 姿势变化复杂，2D 控制图难以稳定。
- 需要批量生产动作和分镜。

3D 基底负责结构，ComfyUI 负责角色身份、服装质感、漫画风格和局部修复。

## 为什么现在先做 role_001 样板

- 先用一个角色验证目录、脚本、manifest、筛选规则是否可用。
- 避免同时处理多个角色导致分类标准混乱。
- 先调好阈值和人工复核流程，再批量推广到其他文件夹。
- role_001 可以覆盖身份、服饰、腿、鞋袜、发型、身材比例等多个模块。

## 为什么不要一开始下载所有素材

- 素材越多，错误分类越多，人工复核成本越高。
- 下载内容可能混入无视频流、重复图、低质量图或不适合训练的素材。
- 当前更需要建立标准和筛选流程，而不是堆数量。
- 先小样本跑通，再扩展到高质量来源。

## 为什么不要训练万能 LoRA

万能 LoRA 会把身份、服装、发型、妆容、姿势和风格绑在一起，后续很难像游戏编辑器一样独立调节。

更稳的路线：

1. 先验证身份候选素材。
2. 再判断是否需要角色身份 LoRA。
3. 服装、腿、鞋袜、发型、妆容先做参考目录和候选集。
4. 只有某个模块需要高频稳定复现时，再单独考虑 LoRA。
