# LoRA 边界

本文件定义不同 LoRA 的职责边界，避免把角色身份、服装、发型、妆容和风格混成一个难以维护的模型。

## 角色身份 LoRA

角色身份 LoRA 负责“林薇是谁”。

它应该学习：

- 脸型。
- 五官比例。
- 黑框眼镜。
- 长深棕色头发的基础身份特征。
- 年龄感和气质。
- 不同角度下仍然稳定的角色辨识度。

它不应该重点学习：

- 某一套固定衣服。
- 某一个场景。
- 某一种姿势。
- 某种一次性妆容或特殊发型。

## 服装 LoRA

服装 LoRA 负责“林薇穿什么”。

它适合学习：

- 固定上衣、外套、裙裤、制服。
- 服装颜色、材质、领口、袖口和层次。
- 鞋包、配饰和成套造型。

它不应该学习角色脸，否则会把服装和身份绑定，导致换装困难。

## 发型 LoRA

发型 LoRA 负责“头发怎么变”。

它适合学习：

- 披发、马尾、盘发、短发等版本。
- 刘海、分缝、发尾卷直、发量轮廓。
- 固定发色或特殊发饰。

如果林薇长期只有一种发型，第一阶段不必单独训练发型 LoRA，可以先由身份 LoRA 和 prompt 控制。

## 妆容 LoRA

妆容 LoRA 负责“脸上的妆感怎么变”。

它适合学习：

- 日常妆。
- 职场妆。
- 正式场合妆容。
- 舞台妆或剧情特殊妆容。

妆容 LoRA 不应该承担角色身份，否则会导致“不上这个妆就不像林薇”。

## 风格 LoRA

风格 LoRA 负责“画面怎么画”。

它适合学习：

- 漫画线条。
- 上色方式。
- 光影质感。
- 镜头语言。
- 分镜整体视觉风格。

风格 LoRA 应尽量使用不绑定单一角色的素材。它的职责是统一漫画画风，而不是固定某个角色。

## 为什么不建议训练万能 LoRA

万能 LoRA 会把多个控制维度绑在一起，看似省时间，实际会降低后期生产自由度。

典型问题：

- 换衣服时脸也变。
- 换发型时妆容和年龄感一起漂。
- 换妆容时角色身份变弱。
- 换场景时旧背景残留。
- 风格权重影响身份稳定性。
- 失败时无法判断问题来自身份、服装、发型、妆容还是画风。

更稳妥的路线是分层资产化：先身份 LoRA，再按需要拆出服装、发型、妆容和风格模块。

## LoRA 与 SD1.5 / SDXL / Pony / FLUX / Qwen / Z-Image 的关系

LoRA 不是跨所有模型通用的外挂。它通常依赖训练时的基础模型架构、参数命名、文本编码器和分辨率生态。

### SD1.5

SD1.5 LoRA 生态成熟、训练成本低、速度快，但基础分辨率和现代画质上限较低。SD1.5 LoRA 通常不能直接用于 SDXL、FLUX、Qwen 或 Z-Image。

### SDXL

SDXL 适合更高分辨率和更稳定的现代图像质量。当前角色身份 LoRA 如果走 SDXL 路线，应使用 SDXL 系 checkpoint 和对应训练配置。

### Pony

Pony 通常属于 SDXL 生态的衍生路线，适合二次元、漫画和角色表现。Pony LoRA 要尽量匹配 Pony 系底模，不要默认拿普通 SDXL LoRA 混用。

### FLUX

FLUX 是不同于 SD1.5/SDXL 的新一代架构生态。FLUX LoRA 需要 FLUX 对应训练和加载方式，不能直接复用 SDXL LoRA。它适合高质量生成和后续路线验证，但当前应先确认本地工作流、显存成本和角色控制能力。

### Qwen

Qwen-Image / Qwen-Image-Edit 更偏向图像生成与指令式编辑生态。即使后续支持 LoRA 或微调，也需要对应 Qwen 架构的训练和加载方式，不能直接使用 SDXL/Pony 的 LoRA。

### Z-Image

Z-Image / Z-Image-Edit 属于新的图像生成与编辑路线。它的 LoRA 或微调方式应按 Z-Image 自身生态处理，不能默认兼容 SDXL/Pony。当前更适合作为后续编辑与对照测试路线，而不是第一阶段身份 LoRA 主线。
